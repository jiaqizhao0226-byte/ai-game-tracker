# 案例分析骨架 & 分类规范

本看板每个案例（`games` 记录）遵循统一的分析骨架与分类维度。录入 / 重构时以本文件为准。

## 一、固定分析骨架（`product_intro` 八段）

`product_intro` 用 Markdown 组织，固定使用以下**加粗小标题**分段。详情页会渲染 `**加粗**` 与 `- 列表`。
标 ✅ 的为必填；无公开信息时写「暂无公开数据」，**不臆造**。数据充分的深案例（如逗逗）八段全填；薄案例只填有内容的段落。

| 段落 | 必填 | 说明 |
|------|:---:|------|
| **一句话定位** | ✅ | 是什么，一句话说清 |
| **核心玩法机制 & AI 介入点** | ✅ | 玩法闭环 + AI 到底做了什么、是否核心（对应 `ai_role`） |
| **数据表现** | ✅ | 增长（MAU/DAU 趋势）、留存、粘性，尽量带**同类对标** |
| **用户画像与核心需求** | | 目标人群、底层需求、典型场景（谁在用、为什么用） |
| **商业化** | | 现状（收入结构）、单位经济（成本/ARPU/CAC）、路径 |
| **技术与数据壁垒** | | 技术链路、独家数据积累、可迁移性/护城河 |
| **风险与局限** | | 已暴露的问题、依赖、争议 |
| **策略启示** | ✅ | 对我们的意义（能不能借鉴 / 要不要警惕）。仅基于已知事实提炼，不夸大 |

> 原则：把已有信息归位到对应段落、把散文里的数字提进「数据表现」；源文没有的段落留空，不补造。团队信息走结构化「核心团队」字段，不占正文段落。

## 二、分类维度取值（枚举，录入必须落在集合内）

### ai_role — AI 介入度（新增核心维度）
- `核心驱动`：AI 即产品本体 / 玩法引擎（AI陪伴、AI原生玩法、生成式UGC）
- `深度增强`：传统玩法，AI 系统性改变体验（AI NPC / AI 队友作为核心系统）
- `外围辅助`：AI 是附加模块，玩法闭环不依赖它（大厅 bot、赛季模式、雇佣型附加玩法）

### gameplay_main → gameplay_sub（大类 / 子类）
- `AI陪伴`：初代陪伴 / 二代陪伴
- `传统玩法+AI`：AI NPC / AI 队友
- `AI原生玩法`：对话模拟类 / 玩法机制类 / 派对游戏 / AI Agent
- `生成式AI驱动UGC`：（暂无细分子类）

### platform — 平台
`PC端` / `移动端` / `网页` / `多平台` / `未知`
（Steam、桌面客户端一律归 `PC端`；跨端归 `多平台`）

### funding_round — 融资 / 资本状态
`未融资` / `天使轮` / `A轮` / `B轮及以上` / `已上市` / `被收购` / `大厂孵化` / `未知`

### region — 区域
`国内` / `海外` / `未知`

### company_size — 团队规模
`大厂` / `创业公司` / `独立开发者` / `未知`

### status — 开发状态
`已上线` / `测试中` / `研发中` / `早期·原型` / `未知`

## 三、已废弃字段
- `gameplay_theme`：72% 为空且与 `gameplay_sub` 重叠，已删除。
- 玩法大类「可交互视频」：已下线，勿再使用。
