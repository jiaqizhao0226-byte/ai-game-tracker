# 案例分析骨架 & 分类规范

本看板每个案例（`games` 记录）遵循统一的分析骨架与分类维度。录入 / 重构时以本文件为准。

## 一、固定分析骨架（`product_intro` 八段）

`product_intro` 用 Markdown 组织，固定使用以下**加粗小标题**分段。详情页会渲染 `**加粗**` 与 `- 列表`。
标 ✅ 的为必填；无公开信息时写「暂无公开数据」，**不臆造**。数据充分的深案例（如逗逗）八段全填；薄案例只填有内容的段落。

| 段落 | 必填 | 说明 |
|------|:---:|------|
| **一句话定位** | ✅ | 是什么，一句话说清 |
| **核心玩法机制 & AI 介入点** | ✅ | 玩法闭环 + AI 到底做了什么、是否核心（对应 `ai_role`） |
| **数据表现** | ✅ | 增长（MAU/DAU 趋势）、留存、粘性，尽量带**同类对标** |
| **用户画像与核心需求** | | 目标人群、底层需求、典型场景（谁在用、为什么用） |
| **商业化** | | 现状（收入结构）、单位经济（成本/ARPU/CAC）、路径 |
| **技术与数据壁垒** | | 技术链路、独家数据积累、可迁移性/护城河 |
| **风险与局限** | | 已暴露的问题、依赖、争议 |
| **策略启示** | ✅ | 对我们的意义（能不能借鉴 / 要不要警惕）。仅基于已知事实提炼，不夸大 |

> 原则：把已有信息归位到对应段落、把散文里的数字提进「数据表现」；源文没有的段落留空，不补造。团队信息走结构化「核心团队」字段，不占正文段落。

## 二、分类维度取值（枚举，录入必须落在集合内）

### 分类两步判据（录入必读）

分类只需回答两个问题，机械可执行：

1. **定骨架**：去掉 AI，这个玩法还成立吗？
   - 成立 → `传统品类+AI`（AI in Game，AI 是叠加）
   - 不成立 → AI Native（进第 2 步，AI 是本体）
2. **Native 内定品类**：有没有要达成的目标 / 结局？
   - 无（关系本身是目的）→ `AI陪伴`
   - 有（生存 / 破案 / 说服 / 推进剧情到结局）→ `AI叙事对话`
   - 其余按体验落到 `AI玩法机制` / `AI Agent(智能体)` / `AI生成UGC`

> 说明：旧的 `ai_role`（核心驱动/深度增强/外围辅助）已删除——它与「骨架」重合（95% 可互推），信息全部并入本判据。

### gameplay_main → gameplay_sub（主品类 单选 / 子类 单选）
大类固定顺序：先 **AI Native**（`AI陪伴` / `AI叙事对话` / `AI玩法机制` / `AI Agent(智能体)` / `AI生成UGC`），后 **AI in Game**（`传统品类+AI`）。首页大类/子类下拉即按此分组排布。

- `AI陪伴`：AI游戏陪玩 / AI伴侣 / AI宠物
- `AI叙事对话`：对话模拟 / 互动叙事　（对话模拟 vs 互动叙事：「不让自由打字还玩得下去吗」——玩不下去=对话模拟，能=互动叙事）
- `AI玩法机制`：AI裁定/主持 / AI生成元素　（按 AI 在机制里的角色划分——裁判/主持:实时评判玩家自由输入定胜负；生成元素:把玩家输入转成可持有的玩法元素(卡/技能/单位)拿去构筑对战。派对属性由 `gameplay_theme` 的「派对游戏」承载）
- `AI Agent(智能体)`：智能体社会（生成沙盒已并入）
- `AI生成UGC`：AI UGC玩法 / 零代码造游戏（当前零代码无在库案例）
- `传统品类+AI`：AI NPC / AI 队友（AI 强化机制而非角色的可留空子类）
- `AI for Game`：游戏基础模型 / 一站式生成平台 / 创意孵化平台 / 创作与分发平台 / 资产与动捕

> **`AI for Game` 是研发侧,独立于上述骨架**。前 6 类都是「玩家侧产品」,适用两条判据;`AI for Game` 收录的是**给开发者用的工具/平台**（生成引擎、资产管线、创作分发平台），玩家不是其用户，故两条判据对它不适用，也不计入 AI Native / AI in Game 的统计。判定：**它的用户是玩家还是开发者？** 是开发者即归此类。**有消费端不影响归类**——创作工具自带内容消费场景是常态（如 Castloop 工具端与消费端并存、Pinoc 既做动捕又有 Game Portal），只要产品叙事的核心是「帮人做出东西」，就归 `AI for Game`。

### scale_tier — 团队规模层级（单值，首页筛选维度）
`大厂/大厂孵化` / `融资创业` / `独立/Indie`

判定顺序（自上而下，命中即止）：
1. **大厂/大厂孵化**：`company_size=大厂`，或 `funding_round` 为 `大厂孵化`/`已上市`。含大厂马甲项目（如崇祯=网易雷火孵化）。
2. **融资创业**：`funding_round` 为 `天使轮`/`A轮`/`B轮及以上`/`被收购`；或虽未披露轮次，但有实据表明是正经公司化运作（拿过种子轮、被孵化、有实体公司与营收规模，如 LoveyDovey 月流水 76 万美元、Aivilization 由港科大孵化）。
3. **独立/Indie**：其余——小工作室、原型作品、众筹/靠爱发电、个人开发者。

> 与 `company_size`/`funding_round` 的关系：后两者是原始字段（可能为「未知」），`scale_tier` 是据其 + `funding_detail`/`team_background` 实据归纳出的**可筛选层级**，用于回答「这是大厂在做还是小团队在做」。

### sort_weight — 手动排序位（可选）
看板默认按 `id` 降序（新收录在前）。补录**老项目**时，`id` 大只代表录入晚、不代表项目新，会不合理地占据前排；此时填 `sort_weight` 指定它该落在的位置（取值参照邻近项目的 id，可用小数插空，如 `31.5`）。不填则回落到 `id`，其余产品顺序不受影响。

### gameplay_facets — 次品类标签（可多值，单选主品类之外的跨类味道）
词表 = 同一套 6 主品类。仅当产品**确实跨类**时加，逗号分隔；主品类单选、facets 多值，二者共同支撑首页「玩法大类」筛选（命中主品类或任一 facet 都算）。
例：Whispers 主 `AI叙事对话` + facet `AI陪伴`；喵呜岛 主 `传统品类+AI` + facet `AI陪伴`。**纪律**：主品类只填 1 个（核心循环），facets 只在某侧面确实很重时加，别 incidental 也挂。

### gameplay_theme — #重点tag（首页显示名；可多值，任何有筛选价值的共性轴，不限于题材）
`多模态玩法` / `腾讯系` / `网易系` / `米哈游系` / `二次元` / `派对游戏` / `推理探案` / `模拟经营` / `AI男友` / `AI女友` / `历史模拟`
多值用逗号分隔（如 `AI女友,米哈游系,多模态玩法`）。取值范围以 [`DashboardClient.tsx`](../src/app/DashboardClient.tsx) 的 `themeOrder` 为准。

判定口径：
- **大厂系**（腾讯系/网易系/米哈游系）：宽口径——出品、孵化、核心团队出身、领投均算（如 Whispers 归米哈游系、Pengu 归腾讯系）。
- **多模态玩法**：**玩家侧**玩法用到文本以外模态的 AI 生成/识别（图像、音乐、视频、视觉理解）。排除两类：①纯语音通道（ASR/TTS 包着文本 LLM，如大厅语音 bot、语音指挥队友）②研发管线的多模态提效（动捕、美术生产管线）。

### platform — 平台
`PC端` / `移动端` / `网页` / `多平台` / `未知`
（Steam、桌面客户端一律归 `PC端`；跨端归 `多平台`）

### funding_round — 融资 / 资本状态
`未融资` / `天使轮` / `A轮` / `B轮及以上` / `已上市` / `被收购` / `大厂孵化` / `未知`

### region — 区域
`国内` / `海外` / `未知`

### company_size — 团队规模
`大厂` / `创业公司` / `独立开发者` / `未知`

### status — 开发状态
`已上线` / `测试中` / `研发中` / `早期·原型` / `未知`
